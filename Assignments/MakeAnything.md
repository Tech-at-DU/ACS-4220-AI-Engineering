# makeanything Project

## Purpose (Why Should I Do This?)

This is your chance to design and build a **portfolio project**—something that solves a real problem and demonstrates everything you've learned. You'll own the entire workflow: from identifying the problem, to shipping with tests, to presenting your thinking. This is the work you'll show potential employers.

## Setup

### Scenario

You've identified a tool that would help your team work better. This is your chance to design and build it from start to finish.

### Solve

#### Identifying a Good Engineering Problem

1. What's annoying about your daily workflow that you wish was easier?
1. What tool or automation would save you the most time?
1. What could you automate to make yourself and others more productive?
1. How could you make teammates' work more fun or relevant?
1. Where can AI-assisted development do something humans can't, and build fast?
    1. _Example_: Parallel workers scaffolding a full-stack app from one spec.
    1. _Example_: A CLI that generates correct, current code snippets for any library.

### First Deliverable

Commit a proposal document describing the problem you'll solve. Put it in a file called `proposal.md` at the project root. Your instructor will look for it there.

## Instructions

1. You need 70%+ to pass.
1. Check the [syllabus](../README.md) for due dates and commit velocity rules.
1. Copy this doc and commit it to your repo. Use the checkboxes to track what you've done.
1. Items marked with :star: are bonus points. Do any, all, or none.

### Requirements

Pick at least 5 of 7:

- [ ] **Project memory** — `CLAUDE.md`, scoped instructions, custom commands.
- [ ] **Spec with teeth** — quality gates and structured acceptance criteria.
- [ ] **Test-first** — commit history shows tests before implementation.
- [ ] **Complexity awareness** — different workflows for different task sizes.
- [ ] **Protocol integration** — external tool servers used in real work.
- [ ] **Scoped rules** — rules that enforce your project's conventions.
- [ ] **Iterative refinement** — evidence of tuning based on output quality.

#### Bonus

- [ ] :star: Design and build a **custom protocol server** for your project's specific needs.
- [ ] :star: Use **parallel workers on branches** to speed up development.
- [ ] :star: Create a **reusable project template** another student could clone and build with immediately.

### Presentation (Day 14 — Wed, May 6)

Each student: 7 minutes + 3 minutes Q&A.

1. **Demo** (3 min) — show it working.
1. **Walkthrough** (4 min) — spec, config, test-first approach. Show how your setup shaped what you built.
1. **Lessons** (3 min) — what worked, what didn't, what you'd change.
1. **Q&A** (5 min) — class and instructor questions.

## Rubric

| | **Needs Improvement** | **Basic** | **Proficient** | **Advanced** |
|:--|:--|:--|:--|:--|
| **Scope & Ambition** | Trivial project or incomplete implementation | Meaningful application demonstrating 3-4 course skills | Meaningful application demonstrating 5+ course skills | Ambitious project with creative application of advanced techniques |
| **Spec Quality** | Missing or vague descriptions | Basic descriptions present | Quality gates with structured, verifiable acceptance criteria | Specs thorough enough that another student could reproduce the build |
| **Workflow Mastery** | Ad-hoc development without consistent process | Some test-first work, inconsistent approach | Full test-first workflow with appropriate complexity awareness | Sophisticated use of design exploration, specialized workers, and parallel execution |
| **Tool Configuration** | Minimal or generic setup | Basic configuration present | Rich, project-specific configuration with evidence of refinement | Config serves as a reusable template for similar projects |
| **Presentation** | Unprepared or unable to explain workflow decisions | Demo functional; basic walkthrough | Clear demo, workflow walkthrough, and lessons learned | Engaging presentation with insights that benefit the whole class |
| **Work Ethic** | < 10 commits, work done last minute | Sporadic commits | Consistent commits across the project period | Iterative refinement visible in commit history |
